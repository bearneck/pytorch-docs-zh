    PyTorch 贡献指南¶

    PyTorch 是一个基于 GPU 加速的 Python 张量计算包，用于使用基于磁带的自动微分系统构建深度神经网络。

    贡献流程¶

    PyTorch 组织遵循 PyTorch 治理 <governance> 原则，技术贡献指南可在 CONTRIBUTING.md 中找到。

    PyTorch 的开发过程包含了核心开发团队与社区之间大量的公开讨论。

    PyTorch 在 GitHub 上的运作方式与大多数开源项目类似。但是，如果您之前从未为开源项目贡献过，以下是基本流程。

    • 确定您要处理的工作。 大多数开源贡献来自于解决个人实际需求。但是，如果您不知道要做什么，或者只是想更熟悉这个项目，以下是一些寻找合适任务的建议：
    • 浏览 问题跟踪器，看看是否有您知道如何修复的问题。已被其他贡献者确认的问题通常更值得研究。我们也维护了一些可能适合新手的标签，例如 bootcamp 和 1hr，尽管这些标签的维护可能不那么完善。
    • 加入我们的 开发讨论区 并告知我们您有兴趣了解 PyTorch。我们非常乐意帮助研究人员和合作伙伴熟悉代码库。
    • 评估您变更的范围，如果变更较大，请在 GitHub issue 上寻求设计意见。 大多数拉取请求都很小；这种情况下，无需告知我们您要做什么，直接开始即可。但如果变更较大，通常最好先通过 提交 RFC 获取一些设计意见。
    • 如果您不确定变更有多大，我们可以帮您评估！只需在 问题 或 开发讨论区 上发帖说明。
    • 某些功能添加非常标准化；例如，许多人向 PyTorch 添加新的算子或优化器。在这种情况下，设计讨论主要归结为"我们是否需要这个算子/优化器？"。提供其有用性的证据，例如在同行评审论文中的使用情况，或在其他框架中的存在，有助于支持您的论点。
      • 添加来自近期发布研究的算子/算法 通常不会被接受，除非有压倒性的证据表明这项新发表的工作具有突破性成果，并最终将成为该领域的标准。如果您不确定您的方法属于哪一类，请在实现 PR 之前先开一个 issue。
    • 核心变更和重构可能很难协调，因为 PyTorch 主分支的开发速度相当快。对于基础性或跨领域的变更，请务必联系我们；我们通常可以就如何将此类变更分阶段进行，使其更易于审查提供指导。
    • 开始编码！
    • 有关以技术形式使用 PyTorch 的建议，请参阅 CONTRIBUTING.md 文件。
    • 开启一个拉取请求。
    • 如果您尚未准备好让拉取请求接受审查，请先创建一个草稿拉取请求 - 之后您可以按"准备就绪以供审查"按钮将其转换为完整的 PR。在草稿阶段，您也可以在 PR 标题前加上 \"[WIP]\"（"进行中"）。我们在进行审查时会忽略草稿 PR。如果您正在进行复杂的变更，最好以草稿形式开始，因为您需要花时间查看 CI 结果以确认是否成功。
    • 为您的变更找到合适的审阅者。我们有一些人会定期查看 PR 队列并尝试审阅所有内容，但如果您碰巧知道受您补丁影响的特定子系统的维护者是谁，可以直接在拉取请求中 @ 他们。您可以了解更多可能审阅您代码的 相关人员。
    • 在拉取请求上迭代，直到它被接受！
    • 我们将尽力减少审查往返次数，并仅在存在重大问题时才阻止 PR。有关拉取请求中最常见的问题，请查看 常见错误。
    • 一旦拉取请求被接受且 CI 通过，您就无需再做其他事情；我们会为您合并 PR。

    入门指南¶

    提出新功能¶

    新功能想法最好在特定的 issue 中进行讨论。请尽可能包含详细信息、任何附带数据以及您提出的解决方案。PyTorch 团队和社区会经常审查新的 issue 和评论，并在他们认为可以提供帮助的地方提供意见。如果您对自己的解决方案有信心，请直接开始实现。

    报告问题¶

    如果您发现了一个问题，请先在仓库的 现有问题列表 中搜索。如果找不到类似问题，请创建一个新问题。请尽可能提供详细信息以重现问题行为。同时，包括任何额外的见解，例如您期望的行为。

    实现功能或修复错误¶

    如果您想修复某个特定问题，最好在该问题下留言说明您的意图。但是，除了我们之前与开发者合作过的情况外，我们通常不会锁定或分配问题。最好在问题下发起讨论，并阐述您提出的解决方案。PyTorch 团队可以提供指导，为您节省时间。

    标记为 [first-new-issue]、`low` 或 [medium] 优先级的问题是最佳切入点，也是开始贡献的好地方。

    添加教程¶

    pytorch.org 上的许多教程都来自社区本身，我们欢迎新的贡献。要了解如何贡献新教程，您可以在此处了解更多信息：`GitHub 上的 PyTorch.org 教程贡献指南 \<https://github.com/pytorch/tutorials/#contributing>`__

    改进文档和教程¶

    我们的目标是提供高质量的文档和教程。偶尔内容中会出现拼写错误或错误。如果您发现可以修复的内容，请向我们提交拉取请求以供考虑。

    请查看 文档 部分以了解我们的系统如何运作。

    参与在线讨论¶

    您可以在 PyTorch 用户讨论论坛 以及面向开发者和维护者的 PyTorch 开发者讨论论坛 上找到活跃的讨论。

    提交拉取请求以修复未解决的问题¶

    您可以在此处查看所有未解决问题的列表 此处。在问题上留言是吸引团队注意的好方法。您可以在此分享您的想法以及计划如何解决问题。

    对于更具挑战性的问题，团队将提供反馈和指导，以帮助您最好地解决问题。

    如果您自己无法修复该问题，留言并分享您是否能重现该问题可以帮助团队识别问题所在。

    审查开放的拉取请求¶

    我们感谢您帮助审查和评论拉取请求。我们的团队努力将开放的拉取请求数量保持在可控范围内，如果需要更多信息，我们会迅速回应，并且我们会合并我们认为有用的 PR。然而，由于关注度很高，额外的审查目光总是受欢迎的。

    提高代码可读性¶

    提高代码可读性对每个人都有帮助。通常，提交少量涉及几个文件的拉取请求比提交一个涉及许多文件的大型拉取请求更好。在 PyTorch 论坛 此处 或与您的改进相关的问题上发起讨论是最好的开始方式。

    添加测试用例以使代码库更健壮¶

    额外的测试覆盖是受欢迎的。

    推广 PyTorch¶

    您在项目、研究论文、文章、博客或互联网上的 general discussions 中使用 PyTorch，有助于提高 PyTorch 和我们不断发展的社区的知名度。如需营销支持，请联系 marketing@pytorch.org。

    问题分类¶

    如果您认为某个问题可以从特定的标签或复杂度级别中受益，请在问题上留言并分享您的意见。如果您认为某个问题分类不当，请留言告知团队。

    关于开源开发¶

    如果这是您第一次为开源项目做贡献，开发过程的某些方面可能对您来说有些陌生。

    • 无法"认领"问题。 当人们决定处理某个问题时，通常想"认领"它，以确保当其他人最终也处理它时不会造成重复劳动。这在开源中效果并不太好，因为某人可能决定处理某事，但最终没有时间去做。请随意以建议的方式提供信息，但最终，我们将依据可运行的代码和粗略的共识来快速推进。
    • 新功能有很高的门槛。 与企业环境不同，在企业中，编写代码的人隐式地"拥有"它，并且可以预期在代码的整个生命周期内维护它；而在开源项目中，一旦拉取请求被合并，它立即成为项目所有维护者的共同责任。当我们合并代码时，我们就是在说，我们这些维护者可以审查后续的更改并对代码进行错误修复。这自然导致了更高的贡献标准。

    需要避免的常见错误¶

    • 您添加测试了吗？ （或者如果更改难以测试，您是否描述了如何测试您的更改？）
    • 我们要求测试有几个动机：
    1. 帮助我们判断以后是否会破坏它
    2. 帮助我们首先判断补丁是否正确（是的，我们确实审查了它，但正如 Knuth 所说，"小心下面的代码，因为我还没有运行它，只是证明了它是正确的"）
    • 何时可以不添加测试？ 有时一个变更无法方便地进行测试，或者变更显而易见是正确的（且不太可能被破坏），那么不进行测试也是可以的。相反，如果某个变更看起来（或已知）很可能被意外破坏，那么花时间制定测试策略就非常重要。
    • 你的 PR 是否过长？
    • 我们更容易审查和合并较小的 PR。审查 PR 的难度与其规模呈非线性增长。
    • 何时可以提交大型 PR？如果在相关 issue 中进行了相应的设计讨论，并获得了将要审查你的 diff 的人员的认可，这将大有帮助。我们也可以就如何将大型变更拆分为可独立交付的部分提供建议。同样，如果 PR 内容有完整的描述也会有所帮助：如果我们知道里面有什么，审查代码会更容易！
    • 对微妙之处添加注释？ 如果你的代码行为有细微差别，请包含额外的注释和文档，以便我们更好地理解你的代码意图。
    • 你添加了一个 hack 吗？ 有时，正确的解决方案是一个 hack。但通常，我们需要对此进行讨论。
    • 你想修改一个非常核心的组件吗？ 为了防止出现重大回归，修改核心组件的 pull request 会受到额外的审查。在进行重大更改之前，请确保你已经与团队讨论过你的更改。
    • 想添加新功能？ 如果你想添加新功能，请在相关 issue 中评论你的意图。我们的团队会尝试对社区进行评论并提供反馈。在构建新功能之前，最好与团队和社区其他成员进行公开讨论。这有助于我们了解你正在做什么，并增加其被合并的机会。
    • 你修改了与 PR 无关的代码吗？ 为了便于代码审查，请在你的 pull request 中仅包含与你的更改直接相关的文件。

    常见问题¶

    • 我如何以审阅者的身份做出贡献？ 如果社区开发者能复现问题、尝试新功能，或以其他方式帮助我们识别或排查问题，这将非常有价值。在任务或 pull request 上评论你的环境详细信息是有帮助的，我们对此表示赞赏。
    • CI 测试失败了，这意味着什么？ 也许你的 PR 是基于一个损坏的主分支？你可以尝试将你的更改变基到最新的主分支上。你也可以在 https://hud.pytorch.org/ 查看主分支 CI 的当前状态。
    • 哪些是最高风险的更改？ 任何涉及构建配置的更改都是高风险区域。除非你事先与团队讨论过，否则请避免更改这些内容。
    • 嘿，我的分支上出现了一个提交，这是怎么回事？ 有时，另一位社区成员会为你的 pull request 或分支提供补丁或修复。这通常是让 CI 测试通过所必需的。

    关于文档¶

    Python 文档¶

    PyTorch 文档是使用 Sphinx 从 Python 源代码生成的。生成的 HTML 被复制到 pytorch.org/docs 主分支的 docs 文件夹中，并通过 GitHub Pages 提供服务。

    • 站点：https://pytorch.org/docs
    • GitHub：https://github.com/pytorch/pytorch/tree/main/docs
    • 服务地址：`https://pytorch.org/docs/main \<https://pytorch.org/docs/main>`__

    C++ 文档¶

    对于 C++ 代码，我们使用 Doxygen 生成内容文件。C++ 文档在特殊服务器上构建，生成的文件被复制到 https://github.com/pytorch/cppdocs 仓库，并通过 GitHub Pages 提供服务。

    • 站点：https://pytorch.org/cppdocs
    • GitHub：https://github.com/pytorch/pytorch/tree/main/docs/cpp
    • 服务地址：https://github.com/pytorch/cppdocs

    教程¶

    PyTorch 教程是用于帮助理解如何使用 PyTorch 完成特定任务或理解更整体概念的文档。教程使用 Sphinx-Gallery 从可执行的 Python 源文件或 reStructuredText (rst) 文件构建。

    • 站点：https://pytorch.org/tutorials
    • GitHub：https://github.com/pytorch/tutorials

    教程构建概述¶

    对于教程，`pull requests \<https://github.com/pytorch/tutorials/pulls>`__ 会触发使用 CircleCI 重新构建整个站点以测试更改的效果。此构建被分片为 9 个工作进程构建，总共需要大约 40 分钟。同时，我们使用 make html-noplot 进行 Netlify 构建，该构建会构建站点但不将 notebook 输出渲染到页面中，以便快速审查。

    PR 被接受后，站点会使用 GitHub Actions 重新构建和部署。
