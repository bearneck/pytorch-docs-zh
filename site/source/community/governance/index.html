    PyTorch 治理 | 运作机制¶

    概述¶

    PyTorch 采用层级化的技术治理结构。

    • 贡献者社区负责提交问题、创建拉取请求，并为项目做出贡献。
    • 一小部分**模块维护者**负责推动 PyTorch 项目的各个模块。
    • 他们由**核心维护者**监督，核心维护者负责把握项目的整体方向。
    • 核心维护者中有一位**首席核心维护者**，作为最终的决策者。

    所有维护者都应高度认同 PyTorch 的设计理念。

    除了维护者之外，我们鼓励社区成员参与贡献、提交问题、提出建议、审查拉取请求并活跃于社区中。只要做出贡献并愿意投入，任何人都可以被接纳为维护者，并获得代码库相应部分的写入权限或所有权。

    技术治理与商业治理严格分离。将技术治理与商业治理分离，确保了任何个人或公司都无法通过"购买"方式影响项目的技术指导。此外，技术治理过程的成员资格属于**个人**，而非公司。也就是说，没有为特定公司保留席位，成员资格与个人而非其雇主公司相关联。

    模块维护者¶

    模块被定义为 PyTorch 组织内的 GitHub 代码库，或核心代码库 pytorch/pytorch 中的目录。每个模块都有其自己的维护者小组。维护者小组负责审查和批准提交、改进设计以及调整模块的范围。每个维护者小组可以采用自己的规则和程序进行决策（默认为多数票决）。模块维护者有权对其他模块维护者做出的决定提出异议------尤其是在该决定影响自身时。当出现争议时，模块维护者小组应提供合理且公开的解释，说明争议点、相关论据以及解决方案。在极少数情况下，如果模块维护者自身无法得出结论，他们将升级至核心维护者进行审查。升级问题由核心维护者根据其规则和程序解决。

    每个维护者小组应为其模块发布公开可用的沟通信息（愿景、大致路线图、设计文档、任何争议及争议解决方案），以便贡献者和其他相关方了解项目的未来方向并参与讨论。

    维护者的职责包括：

    • 处理模块的高优先级问题
    • 处理、审查和合并模块的高优先级拉取请求
    • 维护与模块相关的公开文档
    • 主持公开的开发者会议

    核心维护者¶

    核心维护者应深入理解 PyTorch 代码库和设计理念。他们的职责包括：

    • 阐述项目连贯的长期愿景
    • 以各方都能接受的方式协商和解决争议性问题
    • 接收 PyTorch 利益相关者提出的广泛变更请求，并进行评估/接受（较小的模块级请求由模块维护者处理）

    核心维护者作为一个团体，有权否决模块维护者层面做出的任何决定。核心维护者有权以其认为合适的方式解决争议。核心维护者应公开阐述其决策过程，并为其决策、否决和争议解决方案提供清晰的推理。

    核心维护者是 PyTorch GitHub 组织的管理员，并已列在 Maintainers 页面中。

    首席核心维护者 (BDFL)¶

    可能存在核心维护者无法达成共识的决策。为了做出此类困难决策，核心维护者中会指定并公开宣布一位首席核心维护者，在开源治理模型中通常也称为 BDFL。

    首席核心维护者应公开阐述其决策过程，并为其决策提供清晰的推理。首席核心维护者还负责确认或移除核心维护者。

    提名、确认和移除维护者¶

    原则¶

    • 模块维护者小组的成员资格授予**个人**，基于**功绩**，即他们通过贡献、审查和讨论展现出对组件的深厚专业知识，并且认同该组件如何契合 PyTorch 的整体方向。
    • 要获得维护者小组的成员资格，个人必须展现出与 PyTorch 整体原则的强烈且持续的契合。
    • 模块维护者或核心维护者没有任期限制。
    • 如果维护者长时间不积极参与，则采用宽松的标准将其模块维护状态转为"名誉"状态。每个模块维护者小组可以定义适合该模块的不活跃期限。
    • 成员资格属于个人，而非公司。

    提名流程¶

    • 每个模块有其自己的流程。更多信息请联系模块维护者。但是，如果没有明确的流程，您可以通过提交 此表格 向核心维护者提交请求。核心维护者每三个月会面一次。
    • 如果您向核心维护者提交请求，您的请求信息必须包含以下项目：
    • 被提名人在该模块上代码贡献、代码审查和设计贡献的深度与广度
    • 被提名人与维护者、用户及社区互动的评价（正面与负面）
    • 来自维护者的普遍支持评价
    • 核心维护者随后评估所有信息，并做出最终决定：确认或拒绝提名。核心维护者的决定必须阐述清楚并公开。

    移除流程¶

    • 与提名流程类似，社区中的任何人都可以提名某人从模块维护者或核心维护者职位上移除。
    • 个人也可以自我提名要求移除
    • 核心维护者（排除存在利益冲突的人员）将围绕以下方面请求或收集更多信息：
    • 他们在项目上的活动（或缺乏活动）
    • 他们对相关领域看法的改变，导致与项目整体方向产生冲突
    • 其他使他们不适合担任维护者的信息，例如行为准则问题、他们在项目范围之外但与项目价值观冲突的活动
    • 利益冲突：亲属或恋爱关系
    • 核心维护者随后评估所有信息，并做出最终决定：确认或拒绝移除。核心维护者的决定必须阐述清楚并公开。

    提名核心维护者¶

    • 任何核心或模块维护者都可以提名某人成为核心维护者
    • 首席维护者 (BDFL) 负责评估提名。
    • 首席维护者请求或收集更多关于候选人成为核心维护者实力的信息：
    • 来自其他核心和模块维护者的支持信
    • 来自 PyTorch 社区内利益相关者的普遍支持信
    • 任何适合该候选资格的新相关信息
    • 首席维护者评估所有信息，并做出最终决定：确认或拒绝提名，同时公开清晰地阐述其决策背后的理由。

    移除首席核心维护者及提名新的首席核心维护者¶

    • 核心维护者的绝对多数（75%）可以选择移除首席核心维护者
    • 在首席核心维护者被移除后，或在不可预见的情况下（例如首席核心维护者永久无法履职），核心维护者采用排序复选投票法选举新的首席核心维护者。

    添加、移除和重新界定模块与项目的范围¶

    核心维护者共同负责就 PyTorch 组织中添加、移除和重新界定新模块的范围做出决策，这些模块可以作为 PyTorch GitHub 组织中的新仓库，也可以作为 pytorch/pytorch 仓库中的文件夹。

    他们邀请社区成员（包括他们自己）就此类变更提出建议。 建议是开放式的，但应具备一些基础工作，以便为变更提供令人信服的理由。以下是此流程的一个示例方法：

    1. 访谈研究人员/利益相关者，与社区交流，收集问题；
    2. 阅读论文，参加会议，根据经验构建示例流程；
    3. 评估现状------确保此变更是必要的，例如添加新项目或模块值得付出维护成本；或者移除项目或模块不会从 PyTorch 中移除过多价值；
    4. 创建提案；提案涵盖提案获批后的维护、开发和社区计划。

    核心维护者对提案做出最终决定，并公开阐述决策背后的理由。

    决策制定¶

    无争议的变更¶

    主要工作通过 GitHub 上的议题和拉取请求进行。维护者应避免直接向 PyTorch 仓库推送更改，而应依赖拉取请求。核心或模块维护者批准拉取请求后，无需进一步流程即可合并。列在 Maintainers 页面和 CODEOWNERS 文件中的核心和模块维护者最终批准这些变更。

    就议题或拉取请求通知相关专家非常重要。强烈建议获得给定兴趣领域专家的审查，特别是在拉取请求批准方面。未能这样做可能导致相关专家回滚该变更。

    有争议的决策流程¶

    给定兴趣领域的重大变更需要开启一个 GitHub 议题进行讨论。这包括：

    • 对 PyTorch 框架或库的任何语义或语法更改。
    • 对 Python 或 C++ API 的向后不兼容更改。
    • 对核心框架或库的添加，包括现有库内的重大新功能。
    • 核心功能或平台支持的移除

    核心和模块维护者最终批准这些变更。

    通用项目政策¶

    PyTorch 已成立为 PyTorch a Series of LF Projects, LLC。 适用于 PyTorch 和 PyTorch 参与者的政策，包括商标使用指南，位于 https://www.lfprojects.org/policies/。

    PyTorch 参与者确认，所有新贡献的版权将由版权持有者保留，作为独立的原创作品，且不要求任何贡献者或版权持有者向项目转让版权。除下述情况外，所有对项目的代码贡献必须使用此处提供的 3-Clause-BSD 许可证进行： https://opensource.org/licenses/BSD-3-Clause（\"项目许可证\"）。 所有对外发布的代码都将根据项目许可证提供。 维护者可以基于例外情况，批准对入向或出向贡献使用替代的开源许可证。

    常见问题 ---

    问：如果我希望拥有（或部分拥有）项目的某个部分，例如某个功能领域或领域库，例如 Linear Algebra 或 Torch Vision ，该怎么办？ 这完全有可能。 第一步是开始为现有项目领域做出贡献，并支持其健康与成功。除此之外，您可以通过 GitHub issue 提出新功能或改进项目领域的变更建议。

    问：如果我是希望在内部使用 PyTorch 进行开发的公司，能否被授予或购买董事会席位来推动项目方向？ 不能，PyTorch 项目严格遵循维护者项目理念，并将技术治理与商业治理明确分离。但是，如果您希望参与赞助和支持，可以通过 PyTorch 基金会（PTF）参与其中并进行赞助。您也可以让个别工程师寻求成为维护者，但这并非保证，而是基于功绩的。

    问：PyTorch 项目是否支持资助或支持使用或贡献该项目的独立开发者？ 目前不支持。不过，我们正在研究如何更好地支持围绕 PyTorch 的独立开发者社区。如果您有建议或意见，请在 PyTorch 论坛上联系我们进行讨论。

    问：如何向项目贡献代码？ 如果变更相对较小，可以立即在 GitHub 上开启一个拉取请求，供项目提交者审查和合并。对于较大的变更，请先开启一个 issue 提出建议进行讨论。另请参阅 PyTorch Contributor Wiki 以获取详细的贡献指南。

    问：我能成为项目的提交者吗？ 遗憾的是，目前 PyTorch 的提交过程涉及与 Facebook 基础设施的交互，该过程只能由 Facebook 员工触发。不过，我们正在研究将提交者基础扩展到 Facebook 外部个人的方法，并在工具允许时提供更新。

    问：如果我想在会议或其他场合举办 PyTorch 教程，该怎么办？是否需要\"正式\"成为提交者才能这样做？ 不需要，我们鼓励社区成员在任何可能的时间和地点展示他们的工作。如需营销支持，请联系 marketing@pytorch.org。
